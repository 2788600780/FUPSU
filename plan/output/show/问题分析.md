## 1. IBLT 基础原理

### 1.1 数据结构

一个 IBLT 由 k 个子表组成，每个子表有 ℓ 个 cell，总共 m = k × ℓ 个。每个 cell 存两个值：

```
cell[i][j] = (sum, count)
```

插入元素 x 时，对每个哈希函数 i，算出位置 j = h_i(x)，然后 `sum += x, count += 1`。

### 1.2 一个例子：编码、碰撞、Peeling、Cascade

取 k = 2, ℓ = 3。编码 A = {a, b}，哈希输出：

```
h₀(a)=0, h₁(a)=1
h₀(b)=0, h₁(b)=2          ← a 和 b 在 sub[0][0] 碰撞
```

编码后的 IBLT（空 cell 不标）：

```
         cell 0    cell 1    cell 2
sub[0]  (a+b, 2)           ← 碰撞，count=2
sub[1]            (a, 1)   (b, 1)
```

Peeling：找 count = 1 的 cell，取出元素，删掉该元素的所有副本。

```
(1) sub[1][1] = (a, 1) → peel a
    删除 a: sub[1][1] 清空, sub[0][0] 变成 (b, 1)

(2) sub[0][0] = (b, 1) → peel b
    删除 b: 全部归零。恢复 {a, b} = A。
```

**Cascade 就在 (1)→(2) 之间**：a 和 b 碰撞在 sub[0][0]，count=2 无法直接 peel。但 a 在 sub[1][1] 是纯的——先 peel a，碰撞 cell 的 count 降为 1，b 自动暴露。

### 1.3 核心性质：线性

```
Encode(A ∪ B) = Encode(A) ⊕ Encode(B)
Encode(A \ B) = Encode(A) ⊖ Encode(B)
```

⊕ 和 ⊖ 是逐 cell 的 sum 和 count 加减。这意味着对两个已编码 IBLT 做 cell 级运算，等价于对集合做并集/差集后再编码——不需要先合并集合。

当 m > c_k · n（c₂ ≈ 1.30, c₃ ≈ 1.22）时，peeling 以概率 ≥ 1 − 2^{−λ} 成功恢复全部元素。

## 2. 协议完整流程：三轮实例

### 2.1 初始设定

沿用 §1.2 的 IBLT（k = 2, ℓ = 3）。补充两个新元素的哈希值：

```
a: h₀=0, h₁=1      b: h₀=0, h₁=2      c: h₀=1, h₁=0
d: h₀=2, h₁=2      e: h₀=2, h₁=1

Epoch 0 开始前:
  Party A: X₀ = {a}      K^A_0
  Party B: Y₀ = {b}      K^B_0
```

### 2.2 Epoch 0：初始并集计算

#### Round 1：本地编码

```
Party A:  IBLT_A ← Encode({a})
          sub[0][0]=(a,1), sub[1][1]=(a,1)

Party B:  IBLT_B ← Encode({b})
          sub[0][0]=(b,1), sub[1][2]=(b,1)
```

无网络通信。

#### Round 2：全量 IBLT 交换 + UnionPeel

首次 epoch，交换完整 IBLT（2×3 = 6 个 cell，≈ 72 bytes）。

虚拟并集 IBLT_∪ = IBLT_A ⊕ IBLT_B：

```
IBLT_∪ = Encode({a}) ⊕ Encode({b}) = Encode({a, b})

cell 级:
  sub[0][0] = (a,1) ⊕ (b,1) = (a+b, 2)    ← 碰撞！与 §1.2 完全一致
  sub[1][1] = (a,1)              ← a 的纯 cell
  sub[1][2] = (b,1)              ← b 的纯 cell
```

Peeling（同 §1.2 的两步）：

```
peel sub[1][1] → a → 删除 a → sub[0][0] 从 (a+b,2) 变 (b,1)
peel sub[0][0] → b → 全部清空
```

双方输出 **X₀ ∪ Y₀ = {a, b}**。

#### Round 3：密钥演化

```
Party A:  K^A_1 ← PRF(K^A_0),  CT^A_1 ← Enc(K^A_1, Encode({a})),  擦除 K^A_0
Party B:  K^B_1 ← PRF(K^B_0),  CT^B_1 ← Enc(K^B_1, Encode({b})),  擦除 K^B_0
```

**Epoch 0 结束后**：

```
A: 磁盘 CT^A_1=Enc(K^A_1, {a}), 内存 K^A_1, X₁={a}, peer=Encode({b})
B: 磁盘 CT^B_1=Enc(K^B_1, {b}), 内存 K^B_1, Y₁={b}, peer=Encode({a})
```

---

### 2.3 Epoch 1：增删操作

```
Party A: 添加 c, 删除 a
Party B: 添加 d
```

#### Round 1：本地 IBLT 更新

**Party A**：

```
解密 CT^A_1 → IBLT_A = Encode({a})

添加 c (c ∉ {a}):
  IBLT_A ← IBLT_A ⊕ Encode({c})
  修改: sub[0][1]=(c,1), sub[1][0]=(c,1)

删除 a (a ∈ {a}):
  IBLT_A ← IBLT_A ⊖ Encode({a})
  修改: sub[0][0] 从 (a,1) 变 (0,0), sub[1][1] 从 (a,1) 变 (0,0)

Q_A = 4 个 cell (c 的 2 个 + a 的 2 个)
X₁ = {c}
```

**cell 级展示——sub[0][0] 的变化**：

```
Epoch 0 结束: sub[0][0] = (a, 1)
添加 c:       sub[0][0] 不变（c 的 h₀=1）
删除 a:       sub[0][0] ⊖ (a,1) = (0, 0)    ← 清空
```

**Party B**：

```
解密 CT^B_1 → IBLT_B = Encode({b})

添加 d (d ∉ {b}):
  IBLT_B ← IBLT_B ⊕ Encode({d})
  修改: sub[0][2]=(b+d,2), sub[1][2]=(b+d,2)
         ↑ d 和 b 在 sub[0][2] 和 sub[1][2] 都碰撞了！

Q_B = 2 个 cell
Y₁ = {b, d}
```

**注意**：B 侧 b 和 d 在同一个 IBLT 内碰撞（sub[0][2] 和 sub[1][2] 都 count=2）。这在后续 peeling 中会被 cascade 自然处理，不影响正确性。

#### Round 2：Delta 交换 + Incremental UnionPeel

```
delta_A: 4 个 cell × 12 bytes = 48 bytes
delta_B: 2 个 cell × 12 bytes = 24 bytes
全量交换对比: 72 bytes → delta 交换 72 bytes（这个例子 delta 比例高因为集合太小）
```

虚拟并集：

```
IBLT_∪ = Encode({c}) ⊕ Encode({b, d}) = Encode({b, c, d})
注意 a 不在其中——A 删了 a，B 没有 a。正确。
```

从 Q⁰ = Q_A ∪ Q_B 出发做 Incremental UnionPeel，cascade 自动处理 B 侧 b/d 碰撞。恢复 **{b, c, d}**。

#### Round 3：密钥演化

```
A: K^A_2 ← PRF(K^A_1), CT^A_2 ← Enc(K^A_2, Encode({c})),  擦除 K^A_1
B: K^B_2 ← PRF(K^B_1), CT^B_2 ← Enc(K^B_2, Encode({b,d})), 擦除 K^B_1
```

**Epoch 1 结束后**：

```
A: 磁盘 CT^A_1(已锁), CT^A_2; 内存 K^A_2
B: 磁盘 CT^B_1(已锁), CT^B_2; 内存 K^B_2
```

---

### 2.4 Epoch 2：边界情况

```
Party A: 删除 a（已在 Epoch 1 删了！）
Party B: 删除 b, 添加 e
```

#### Round 1

**Party A**：

```
解密 CT^A_2 → IBLT_A = Encode({c})    ← a 确实不在

删除 a: a ∈ {c}? 否 → no-op
  IBLT_A 不变, Q_A = ∅
```

**如果不检查**：执行 `IBLT_A ⊖ Encode({a})` → sub[0][0] count 从 0 变 -1, sub[1][1] count 从 0 变 -1。sum 变成负值 -a。后续 peeling 中负 count 破坏所有经过这些 cell 的元素的恢复。**这就是论文 Alg.1 中检查的必要性。**

**Party B**：

```
解密 CT^B_2 → IBLT_B = Encode({b, d})

删除 b (b ∈ {b,d}):
  IBLT_B ⊖ Encode({b})
  修改: sub[0][0] 从 (b,1) 变 (0,0), sub[0][2] 从 (b+d,2) 变 (d,1)
         ↑ cascade: b 被删后 d 在 sub[0][2] 暴露

添加 e (e ∉ {b,d}):
  IBLT_B ⊕ Encode({e})
  修改: sub[0][2] 从 (d,1) 变 (d+e,2), sub[1][1] 从 (0,0) 变 (e,1)
         ↑ d 和 e 新碰撞

Y₂ = {d, e}, Q_B = 4 个 cell
```

#### Round 2

```
delta_A: 0 bytes（A 无有效修改）
delta_B: 4 个 cell × 12 bytes = 48 bytes

虚拟并集: Encode({c}) ⊕ Encode({d, e}) = Encode({c, d, e})
b 不在并集中——B 删了 b, A 也没有 b。正确。
```

Incremental UnionPeel 恢复 **{c, d, e}**。

#### Round 3

```
A: K^A_3 ← PRF(K^A_2), CT^A_3 ← Enc(K^A_3, Encode({c})),   擦除 K^A_2
B: K^B_3 ← PRF(K^B_2), CT^B_3 ← Enc(K^B_3, Encode({d, e})), 擦除 K^B_2
```

## 3. 逐轮密钥状态追踪

以下跟踪 A 的所有密钥、密文、以及攻击者在不同时间点攻破所能恢复的内容。

```
         Epoch 0           Epoch 1           Epoch 2          [攻击]
         │                 │                 │                  │
X_A:  {a}         →    {c}         →    {c}
         │                 │                 │
密钥链:  K⁰→K¹            K¹→K²            K²→K³
擦除:    K⁰ ✗              K¹ ✗              K² ✗
         │                 │                 │
密文:   CT¹               CT²               CT³
      =Enc(K¹,          =Enc(K²,          =Enc(K³,
      {a})              {c})              {c})
         │                 │                 │                  │
         └─────────────────┴─────────────────┴────────┬─────────┘
                                                      │
                                         攻击者攻破 A, 拿到:
                                           K^A_3
                                           CT^A_1, CT^A_2, CT^A_3 (磁盘)
                                           所有网络通信记录
```

**攻击者能做什么**：

```
K^A_3 已知
→ 解密 CT^A_3 → Encode({c})
→ Peel → 恢复 X₃ = {c}
✓ 当前集合不可避免地泄露

K^A_3 → PRF(K^A_3) → K^A_4 → K^A_5 → ...
✓ 未来密钥可推算（forward-only, 不是 post-compromise）
```

**攻击者不能做什么**：

```
要解密 CT^A_2，需要 K^A_2。
K^A_3 = PRF(K^A_2, "fpsu-evolve-A")
→ 从 K^A_3 逆推 K^A_2 = 暴力搜索 2^256 → 不可行（PRF 单向性）
→ CT^A_2 解不开 → Encode({c}) 拿不到 → X₂ 不可知

同理：
→ K^A_2 不可知 → K^A_1 不可知 → CT^A_1 解不开 → X₁ 不可知
```

**攻击者从网络记录能推断什么**：

```
Epoch 0: 观察到全量 IBLT 交换的字节数 → 推断初始集合大小 ≈ |X₀ + Y₀|
Epoch 1: 观察到 delta_A = 48 bytes → 推断 A 有 4 个 cell 被修改
         （下界定理：增删个数是不可避免的泄漏）
Epoch 2: 观察到 delta_A = 0 bytes → 推断 A 本轮无有效修改

但攻击者不知道:
  - 具体哪些元素被增删
  - 历史集合 (X₁, X₂) 的内容
  - a 是在 epoch 1 删的还是 epoch 2 删的（只能通过 delta 大小猜，无法确证）
```

## 4. 两个核心问题的形式化分析

### 4.1 问题一：添加已有元素会怎样？

**场景**：A 的当前集合 X = {a, b}，A 想"添加" a。

**如果跳过检查**：

```
IBLT_A = Encode({a, b}) ⊕ Encode({a}) = Encode({a, a, b})

a 被编码了两次：
  在 a 的每个位置 j = h_i(a):
    操作前: sum = a + (其他元素贡献), cnt = 1 + (其他)
    操作后: sum = 2a + (其他), cnt = 2 + (其他)
    → a 的虚拟 cnt ≥ 2 → 不纯 → 无法被 peel
    → a 永久卡在 IBLT 里 → 并集输出缺少 a → 正确性被破坏
```

**正确做法（论文 Alg.1 已明确）**：

```
for each x ∈ Δ⁺X_t:
    if x ∉ X_{t-1}:
        IBLT ← IBLT ⊕ Encode({x})
    else:
        skip  ← 幂等操作：添加已有元素是 no-op
```

**为什么不需要和对方判定**：添加是纯本地操作。IBLT_A 的职责是表达 A 的集合，不是表达并集。A 集合中的元素在 IBLT_A 中各出现恰好一次。至于这个元素是否也在 B 的集合中，由 Round 2 的 UnionPeel 通过 OT+OPRF 处理。

### 4.2 问题二：删除前需要和对方判定吗？

**场景**：A 要删除 x，且 x ∈ X。

**子问题 2a**：如果 x 也在 Y 中（交集元素），A 删了 x 后，并集里 x 还会在吗？

```
Epoch 1: A 删 x, B 不动

Round 1 (A 本地):
  IBLT_A = Encode(X_{t-1}) ⊖ Encode({x})
  现在 IBLT_A 里没有 x 了

Round 2 (UnionPeel on IBLT_A ⊕ IBLT_B):
  对于 x 的每个位置 j = h_i(x):
    cnt_A[j] = 0 (A 删了)
    cnt_B[j] = 1 (B 有 x)
    虚拟 cnt = 1 → 纯 cell → peel 出 x
  → x 出现在并集中 ✓

这是正确行为：并集 = 任何一方有就出现。
A 删除只意味着 A 不再持有 x；B 还有，所以 x 仍在并集中。
```

**子问题 2b**：那能不能"既然 B 有 x 我就不删了"？

```
假设 Epoch 1: A 不删 x（因为"B 也有"）

X₁ 仍 = {..., x, ...}（错误！A 实际上不想要 x 了）

Epoch 2: A 再次收到"删除 x"的指令
  检查: x ∈ X₁? 是 → 执行删除 → 但这是基于一个已被污染的 X₁
  → 如果 epoch 1 到 epoch 2 之间 x 被用于其他计算（如交集大小估计），
    错误会传播

更根本的问题：X_t 必须正确反映 A 在第 t 轮的集合。
集合语义要求：如果 A 想删 x，X 里就不应有 x。
IBLT 不变量要求：IBLT_A = Encode(X_A)。
这两个不变量必须逐轮维护，不能因为"并集结果不变"就跳过更新。
```

**子问题 2c**：有前向隐私问题吗？

```
攻击者在 epoch τ 攻破 A，拿到 K^A_τ 和 X_τ。

攻击者看到 X_τ 中不包含 x。
→ 能推断"x 在 τ 之前某个时候被删了"吗？
→ 不能。因为：
  1. 先验上，x 可能从来没在 X 中出现过
  2. 要验证 x 的历史存在性，需要解密历史密文 CT_{<τ}
  3. PRF 单向性保证无法从 K^A_τ 推出 K^A_{<τ}
  4. 历史密文全部不可解密 → 历史集合内容对攻击者透明
```

**为什么不需要和对方做判定——总结**：

| 理由 | 说明 |
|------|------|
| IBLT 忠实性 | IBLT_A 必须 = Encode(X_A)。集合变了 IBLT 必须变，不论对方有什么 |
| UnionPeel 自动处理 | 虚拟并集 = IBLT_A ⊕ IBLT_B，并集语义自动处理独有/交集元素 |
| 下界定理 | 增删个数的泄漏是不可避免的（数学上可证），但集合内容是受保护的 |
| 前向隐私 | PRF 链的单向性保证了历史密文不可解密 |
| 设计原则 | Round 1 = 纯本地（维护 IBLT 不变量），Round 2 = 纯协作（安全计算并集） |

## 5. 完整不变量总结

F-PSU 协议的正确性建立在以下不变量上，它们在每轮结束时必须成立：

```
1. 集合忠实性:
   IBLT_A^{(t)} = Encode(X_t)   每个元素恰好出现一次
   IBLT_B^{(t)} = Encode(Y_t)

2. 密钥单向性:
   K_{t+1} = PRF(K_t, label)
   K_t 在生成 K_{t+1} 后立即擦除

3. 密文一致性:
   CT_{t+1} = Enc(K_{t+1}, IBLT^{(t)})
   存储的是完整 IBLT，不是增量

4. Delta 检查（追加，Alg.1 已更新）:
   添加: x ∉ X_{t-1} → 执行；x ∈ X_{t-1} → no-op
   删除: x ∈ X_{t-1} → 执行；x ∉ X_{t-1} → no-op

5. Cascade 保证（Lemma 1）:
   从 k|Δ| 个 delta cell 出发，传播 ≤ 2·k|Δ| 个 cell
```

违反任何一条都会导致正确性或安全性失败：

| 违反 | 后果 |
|------|------|
| (1) 重复编码 | 元素 count ≥ 2 → 无法 peel → 并集缺失元素 |
| (1) 遗漏删除 | IBLT 与集合不一致 → 后续 epoch 累积错误 |
| (2) 未擦除旧密钥 | 攻击者攻破后可回溯解密 → 前向隐私丧失 |
| (3) 存储增量 | 无法直接恢复当前集合 → 需要从 epoch 0 递推 |
| (4) 未检查重复/缺位 | 重复编码或幽灵删除 → 正确性失败 |

## 6. 加法优化方案：小 IBLT + Mini UnionPeel 的形式化分析

> 2026-05-16 与刘老师讨论后新增

### 6.1 动机

当前协议（§2.3）中对加法的处理方式为：
- Round 1: 将 Δ⁺ 编码进大 IBLT（修改 `k|Δ^+|` 个 cell）
- Round 2: 从 `k|Δ^+|` 个 modified cell 出发做 UnionPeel，加上 cascade ≤ `2k|Δ^+|`

问题：UnionPeel 的初始查询集 `Q^{(0)}` 混入了加法 cell，这些 cell 需要在**稀疏大表**上执行 OT+OPRF 逐 cell 查询。

优化思路：加法产生的是**新元素**，天然构成一个独立的小集合。将这个集合编码为**紧凑小 IBLT**，做一次**全新剥离**（fresh peel），比在大表上稀疏查询更高效。

### 6.2 混合加法协议（形式化）

**Round 1 修改**：

```
Party P₀:
  解密 IBLT_C ← Dec(K_{t-1}^C, CT_stored_C)

  // 删除：保持原方案（纯本地减法）
  for each x ∈ Δ⁻X_t:
    if x ∈ X_{t-1}:
      IBLT_C ← IBLT_C ⊖ Encode({x})
      Q_del ← Q_del ∪ {(i, h_i(x)) : i ∈ [k]}

  // 加法：编码到独立小 IBLT（不合并进大 IBLT）
  S_Δ ← Δ⁺X_t \ X_{t-1}                           // 本端净新增
  IBLT_ΔX ← Encode(S_Δ)                            // 小 IBLT, m_Δ = k·ℓ_Δ
                                                     // ℓ_Δ ≥ c_k · (|S_Δ| + |T_Δ|)

  X_t ← (X_{t-1} ∪ Δ⁺X_t) \ Δ⁻X_t
```

P₁ 对称操作。**零额外通信**（Round 1 仍为纯本地）。

**Round 2 修改**：

```
Round 2a ── Mini UnionPeel ──
  Π_UnionPeel(IBLT_ΔX, IBLT_ΔY)
  → V_Δ⁺  (双方获知本轮并集新增元素)
  
  通信: O(c_k · k · |Δ⁺| · (σ + κ)) bits
  对比原方案: O(k · |Δ⁺| · (σ + κ)) bits（仅初始）
                      + O(cascade_cost)（不可忽视）

Round 2b ── Merge & Main UnionPeel ──
  IBLT_C ← IBLT_C ⊕ Encode(S_Δ)                    // 补合并入大 IBLT
  Q^{(0)} ← Q_del                                    // 仅删除修改的 cell
  
  Π_UnionPeel(IBLT_C, IBLT_S) starting from Q^{(0)}
  → V_main

  V ← V_Δ⁺ ∪ V_main
```

### 6.3 正确性证明

**Claim 1 (Mini UnionPeel 正确性)**。`IBLT_{Δ X} ⊕ IBLT_{Δ Y} = Encode(S_Δ ∪ T_Δ)`，对紧凑小 IBLT 执行标准 UnionPeel 以概率 `≥ 1 − 2^{−λ}` 恢复 `S_Δ ∪ T_Δ`。

**Claim 2 (主 UnionPeel 正确性不变)**。主 UnionPeel 从 `Q_{del}` 出发，只处理由删除引起的 cascade。由于 `S_Δ` 和 `T_Δ` 在 mini peel 阶段已被独立恢复，且在此后本地合并入大 IBLT，主 peel 的开始状态等价于原方案。

**Claim 3 (无重复、无遗漏)**。`V = V_{Δ^+} ∪ V_{main}` 等价于原方案计算的并集，因为：
- `V_{Δ^+}` 包含所有仅由新增产生的并集元素
- `V_{main}` 包含所有原有并集元素（含因删除而暴露的 intersect 元素）
- 两集合不相交：`V_{Δ^+} ∩ V_{main} = ∅`（新元素不可能在原来的大 IBLT 中）

### 6.4 通信复杂度对比

设 `d^+ = |Δ^+ X| + |Δ^+ Y|`, `d^- = |Δ^- X| + |Δ^- Y|`, `d = d^+ + d^-`。

**Cell 查询量对比**（每 cell 包含一次 OT + 一次 OPRF 查询）：

| | 原方案 | 优化方案 |
|---|---|---|
| 加法 cell 数 | `[k, 3k] · d^+` | `c_k · k · d^+` **（新鲜剥离，0 cascade）** |
| 删除 cell 数 | `[k, 3k] · d^-` | `[k, 3k] · d^-` **（不变）** |

对于 `k=3`, `c_3 ≈ 1.222`，取典型 cascade 系数 1.5（介于无 cascade 和最大 cascade 之间）：

| | 原方案 | 优化方案 | 节省 |
|---|---|---|---|
| `d^+ = d^- = d/2` | `4.5d + 4.5d = 9d` | `3.67d^+ + 4.5d^- = 8.17d` | **9.3%** |
| 加法为主 (`d^+ ≫ d^-`) | `4.5d^+ + ε` | `3.67d^+ + ε` | **18.4%** |
| 最大 cascade | `9d^+ + 9d^- = 18d` | `3.67d^+ + 9d^-` | **显著** |

**关键增益不是平均节省，而是方差缩小**：原方案中加法 cell 的 cascade 可能穿过删除 cell 的交叉区域，导致不可预期的长 cascade。优化方案将加法剥离到小 IBLT 做 fresh peel——通信量是确定性的 `c_k · k · d^+`，无随机因素。

### 6.5 支持论证

**为什么 Mini UnionPeel 优于稀疏查询**：

1. **结构分离**。加法和删除在 IBLT 中的行为截然不同：加法 create singletons（cnt 0→1），删除可能 create singletons（cnt 2→1）或 destroy singletons（cnt 1→0）。将加法分离到小 IBLT 避免了两种行为在同一个 UnionPeel 中纠缠。

2. **稠密 vs 稀疏**。小 IBLT 是稠密的（所有 `kℓ_Δ` cell 都与 `Δ^+` 直接相关），UnionPeel 对稠密表的效率天然高于对大表的稀疏跳跃查询。

3. **零 cascade 保证**。小 IBLT 是 fresh peel——剥完就干净。不像稀疏查询，cascade 可能穿过加法 cell 和删除 cell 的交叉区域，传播到意想不到的位置。

4. **实现简洁**。无需修改 UnionPeel 子协议本身（复用 PT26），只需在 Round 1-2 之间插入一步，架构改动最小。

5. **下界兼容**。泄漏仍为 `|Δ^+|, |Δ^-|` 四项计数，不增不减，保持 leakage-optimal 性质。

## 7. 两种删除方案的对比分析

### 方案 A：逐元素 OT/OPRF 判定

```
P0 要删 x:
  先和 P1 做 OT/OPRF 判定 P1 是否有 x
  ├─ P1 没有 → P0 本地减, P1 得知"P0 删了 x"
  └─ P1 有   → P0 不减, P0 得知"P1 有 x"
```

### 方案 B：本地 IBLT 减法（当前方案）

```
P0 要删 x:
  if x ∈ X_{t-1}:
    IBLT_C ← IBLT_C ⊖ Encode({x})    // 纯本地, 0 通信
```

### 对比分析

| | 方案 A (OT/OPRF) | 方案 B (IBLT 减法) |
|---|---|---|
| 通信量 | `|Δ⁻|·(σ+κ)` bits | **0** |
| 交互轮数 | ≥1 | **0** |
| P1 学到什么 | `|Δ⁻X|` + 每个被删元素的**身份** | 仅 `|Δ⁻X|`（不可避免的下界） |
| P0 学到什么 | 并集 + "P1 有没有我想删的每个 x" | 仅并集 |
| 不变量 | ✗（P1 有 x 时不减 → IBLT ≠ Encode(X)） | ✓ |

**结论：方案 B 严格更优。** 三条理由：

1. **通信**：方案 A 把零通信操作变成了 `|Δ⁻|` 次 OT/OPRF。IBLT 的代数结构天然支持本地减法——`Encode(A \ B) = Encode(A) ⊖ Encode(B)`——没有理由为此增加通信。

2. **泄漏**：下界定理证明 `|Δ⁺|, |Δ⁻|` 四项计数是不可避免的。方案 B 恰好泄漏这四项（leakage-optimal）。方案 A 额外泄漏了删除元素的**身份**和**交叉成员信息**，破坏了最优性。

3. **不变量**：方案 A 中"P1 有 x 所以 P0 不减"导致 `IBLT_C ≠ Encode(X_C)`。集合语义被对方数据绑架——"我想删什么取决于你有没有"——这在协议设计中不可接受。

### 删除后的并集计算

两种方案在删除后都要跑 UnionPeel 求并集。区别在于进入 UnionPeel 时的 IBLT 状态不同。

**方案 A 删除后的 IBLT 状态**：P0 只删了 P1 没有的那些元素。P1 有的交集元素被保留了。

```
例: X = {1,2}, Y = {2,3}, P0 要删 {1,2}

方案 A 后:
  IBLT_C = Encode({2})      // 删了 1 (P1没有), 保留了 2 (P1有)
  IBLT_S = Encode({2,3})    // 不变
  UnionPeel → 并集 = {2,3}  ✓ (1 确实消失了, 2 因 P1 有而保留)
```

**方案 B 删除后的 IBLT 状态**：P0 忠实删了所有要删的。

```
同一例:
  IBLT_C = Encode({})       // 1 和 2 都删了
  IBLT_S = Encode({2,3})    // 不变

  Q_del = 2 在 IBLT_C 中的 k 个 cell
  UnionPeel 从 Q_del 出发:
    2 的虚拟 cell: IBLT_C 侧 cnt=0, IBLT_S 侧 cnt=1 → singleton → peel 出 2
    1 的虚拟 cell: 两侧 cnt 都为 0 → 无输出
  并集 = {2,3}  ✓
```

**关键洞察**：方案 B 虽然多删了一个元素（2），但 2 在 P1 那边还有一份，虚拟并集中 cnt=1 → 仍然会被 peel 出来。方案 A 通过"不删"来保留 2，方案 B 通过"删了但 P1 的副本让它自动复活"来达到同样效果。方案 B 更干净——P0 不需要知道 P1 有什么，IBLT 代数自动补偿。

| | 方案 A | 方案 B |
|---|---|---|
| 删除后 IBLT 状态 | 取决于 P1 有什么 | 仅取决于 P0 自己的集合 |
| 交集元素去向 | P0 不减 → 虚拟 cnt=2 → peel 两份 | P0 减了 → 虚拟 cnt=1 → peel 一份 |
| 最终并集 | 相同 | 相同 |
| 信息需求 | P0 需知道 P1 有没有 x | P0 不需任何对端信息 |
