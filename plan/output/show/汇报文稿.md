## 1. 背景与问题

### 1.1 什么是 PSU？

**Private Set Union（隐私集合并集）**：两方各自持有私有集合，协作计算并集，不泄露各自元素。

```
          Alice                          Bob
    X = {a, b, c}                  Y = {c, d, e}
          │                              │
          └──────────  PSU  ─────────────┘
                      │
                 X ∪ Y = {a, b, c, d, e}
```

应用场景：隐私身份管理、联合风控、隐私数据库联合查询、联邦学习。

### 1.2 现状与契机

| 时间 | 进展 |
|------|------|
| 2019 以前 | PSU 比 PSI 慢 30 倍以上 |
| Eurocrypt 2026 | **Piske & Trieu (PT26)**：IBLT-based PSU, 2^20 元素 < 3 秒,首次追平 PSI |

核心突破——**Union Peelability**：IBLT 线性性质允许在两方各自持有的 IBLT 上直接做虚拟并集的 peeling。

### 1.3 但是：它是静态的

```
静态 PSU 的痛点：
┌─────────────────────────────────────────────────────────┐
│  每次计算都要重新编码整个集合                                │
│  改 10 条数据 → 重传 100 万条 → 浪费 10 万倍的计算和通信       │
└─────────────────────────────────────────────────────────┘
```

现实场景中数据持续演化——黑名单每天增删、联系人列表每周更新。每次全量重算效率极低。

## 2. 核心问题

```
┌──────────────────────────────────────────────────────────────┐
│   能不能让 PSU 同时满足两个性质：                                 │
│                                                              │
│   ① Updatable（可更新）                                        │
│      每轮代价 ∝ |Δ|（更新量），而非 |X|（总数据量）                 │
│                                                              │
│   ② Forward-Private（前向隐私）                                │
│      密钥泄露后，历史并集结果不可恢复                             │
└──────────────────────────────────────────────────────────────┘
```

**赛道状态**：
- 可更新 PSI：有人做了（FUPSI, WZC+24），但用 PIR 路线，非 PSU
- 可更新 PSU：**目前没有任何方案存在**
- 前向隐私 + 可更新 PSU：**从来没人提过**

**根本性障碍**：可更新协议中消息长度本身就是泄漏信道——update 消息大小暴露增删元素数量。这个问题能否避免？§4 给出答案。

## 3. 技术方案：F-PSU 协议

### 3.1 协议全景

```
Epoch t 的三轮结构：

  Round 1                Round 2               Round 3
  ┌──────────┐          ┌──────────┐          ┌──────────┐
  │ 本地 IBLT │    Δ     │ UnionPeel│  并集    │ 密钥演化  │
  │ 增量更新  │ ──────► │ (增量版) │ ──────► │ 安全擦除  │
  └──────────┘          └──────────┘          └──────────┘
    O(k|Δ|)              O(k|Δ|) OT + OPRF     O(1) PRF
```

### 3.2 创新一：Delta IBLT —— 如何实现可更新？

**IBLT 编码的线性性质**：

```
  Encode(A ∪ B)  =  Encode(A)  ⊕  Encode(B)    （cell-wise 加）
  Encode(A \ B)  =  Encode(A)  ⊖  Encode(B)    （cell-wise 减）
```

**这意味着什么？** 每轮不需要重建整个 IBLT，只需在上轮基础上增量：

```
  Epoch t-1 结束时：  IBLT_C ← Encode(X_{t-1})    [已加密存储]

  Epoch t 开始时：
    1. 解密 IBLT_C
    2. IBLT_C ← IBLT_C  ⊕  Encode(Δ⁺X_t)    ← 加入新增
                IBLT_C  ⊖  Encode(Δ⁻X_t)    ← 减去删除
    3. 用新密钥重新加密

  仅触碰 k|Δ| 个 cell —— 与总集合大小 |X| 无关
```

```
  ┌─────────────────────────────────────┐
  │        Delta IBLT 更新示意            │
  │  IBLT (m = kℓ cells)                │
  │  ┌───┬───┬───┬───┬───┬───┬───┬───┐ │
  │  │   │ █ │   │   │ █ │   │   │ █ │ │  █ = 被 Δ 修改的 cell
  │  └───┴───┴───┴───┴───┴───┴───┴───┘ │  = k|Δ| 个
  │  更新只碰 k|Δ| 个 cell               │
  └─────────────────────────────────────┘
```

### 3.3 创新二：Incremental Peeling —— 如何更快地 peeling？

**观察**：只有被 delta 碰过的 cell 的 count 值才可能改变。其余 cell 上轮什么状态，这轮还是什么状态。

```
  全量 UnionPeel（静态 PSU）：扫描所有 kℓ 个 cell
  增量 UnionPeel（F-PSU）：只从 k|Δ| 个修改 cell 出发 + cascade 传播

  n = 2^16, |Δ| = 2^12:  全量 ≈ 197K cells → 增量 ≈ 33K cells → 6× 减少
```

**Cascade Bound Lemma**：以压倒性概率（≥1−2^{−λ}），增量 peeling 从 k|Δ| 个修改 cell 出发，传播不超过 2·k|Δ| 个 cell，O(log k|Δ|) 轮内终止。实测 cascade ≈ 1.95 × k|Δ|，100% 成功率。

### 3.4 创新三：PRF 密钥链 —— 如何实现前向隐私？

**零额外通信的前向隐私**：

```
  密钥链：  K₀ ──PRF──► K₁ ──PRF──► K₂ ──PRF──► K₃ ──PRF──► ...

  每轮操作：  K_t ← PRF(K_{t-1})
             CT_t ← ChaCha20.Enc(K_t, IBLT_t)
             安全擦除 K_{t-1}
```

```
  为什么是前向安全的？

  攻击者在 epoch τ 拿到 K_τ：
    K_τ  已知 ──► 可以解密 CT_τ（当前轮，无可避免）
    K_{τ-1} 不可知 ←── PRF 是单向的，无法从 K_τ 倒推
    → CT_{τ-1} 解不开 → CT_{τ-2} 解不开 → ...

  历史数据全部安全。成本：每轮 1 PRF + 1 ChaCha20，无额外通信。
```

## 4. 理论贡献：泄漏下界定理

### 4.1 核心问题：可更新 PSU 能不泄露任何信息吗？

能不能把增删个数也藏起来，让攻击者只知道集合大小变了，但不知道变了几条？

### 4.2 答案：不能

**定理（Efficiency-Leakage Trade-off）**：

```
  任何正确且半诚实安全的 updatable PSU 协议，若其每轮通信量为
  O(|Δ|)，则必然泄露每方的增删个数：
      (|Δ⁺X|, |Δ⁻X|, |Δ⁺Y|, |Δ⁻Y|)

  若想隐藏这些值，通信量必为 Ω(N·σ) —— 无法做到 updatable。
```

### 4.3 证明思路（三步归谬）

```
  假设：协议不泄露增删个数
  ─────────────────────────
  │ 所有 update 消息长度必须一致
  └──────────┬───────────────
             ▼
  ─────────────────────────
  │ 小 update 也要发 Ω(Nσ) 比特（信息论：传 N 个元素需要这么多比特）
  └──────────┬───────────────
             ▼
  ─────────────────────────
  │ 与 updatable 定义矛盾（updatable 要求通信 ∝ |Δ|，而非 N）
  └────────────────────────
```

**核心洞察**：消息长度是半诚实模型下必然可观察的量。你要 updatable → 消息长度随 |Δ| 变化 → 观察消息长度就学到了 |Δ|。这不是工程取舍，是**数学上不可避免的**。

### 4.4 F-PSU 是最优的

F-PSU 恰好泄漏这四个数，一比特不多。在 1-updatable 协议类中，F-PSU 是 **leakage-optimal**。

| | 泄漏内容 | 是否可避免 |
|---|---|---|
| F-PSU 泄漏 | \|Δ⁺X\|, \|Δ⁻X\|, \|Δ⁺Y\|, \|Δ⁻Y\| | 不可避免（本定理） |
| 额外泄漏 | 元素身份、历史集合、交集大小 | 全部避免 |

## 5. 安全性

**威胁模型**：半诚实 + 自适应腐化 | **安全定义**：UC 框架，(F_OPRF, F_OT)-hybrid | **假设**：ChaCha20 PRG, HMAC-SHA256 PRF

```
  ┌─────────────────────┐     ┌─────────────────────┐
  │  Per-Epoch Security  │     │  Key Management      │
  │  (Round 1-2)         │  +  │  (Round 3)           │
  │  UC-realizes F_psu   │     │  PRF chain → F_fwKey │
  └──────────┬───────────┘     └──────────┬──────────┘
             └──────────┬─────────────────┘
                        ▼
              ┌───────────────────┐
              │  UC Composition   │
              │  → UC-realizes    │
```

模块化分解的实用价值：per-epoch PSU 可独立升级（→ ePSU 得执行内保护, → PuGT26 得恶意安全），密钥管理无需修改，前向隐私自动继承。

### 6.6 OT/OPRF 集成开销实测

KOS OT extension 集成到协议循环中，throughput 维持 **5.2M OTs/s**（匹配 standalone benchmark，零性能损失）。

| n | \|Δ\| | OT ext (ms) | OT+OPRF 占 epoch>0 比例 |
|---|------|------------|----------------------|
| 16K | 1K | 1.7 | 11.2% |
| 65K | 4K | 6.3 | 3.4% |
| 262K | 16K | 25.2 | 0.7% |
| 1M | 65K | 101 | 0.07% |

**结论**：OT/OPRF 不是瓶颈——对于实际规模的集合（n ≥ 16K），占比 < 12%。IBLT peeling 是真正的计算瓶颈。

## 7. 扩展方向

### 7.1 通用增量 IBLT 框架：PSU → PSI → PSR

```
  ┌──────────────────────────────────────────────────┐
  │         Incremental IBLT Maintenance              │
  │         (Delta IBLT + Incremental Peeling)        │
  │                                                   │
  │    ┌─────────┐  ┌─────────┐  ┌─────────────────┐ │
  │    │   PSU   │  │   PSI   │  │   PSR (Private  │ │
  │    │ (本工作) │  │ (直接    │  │   Set Reconci-  │ │
  │    │         │  │  派生)   │  │   liation)      │ │
  │    └─────────┘  └─────────┘  └─────────────────┘ │
  └──────────────────────────────────────────────────┘
```

**PSI**：UnionPeel 恢复 X∪Y → 去除各自已知集合 → X∩Y。可同时获得 updatable PSI + 前向隐私（填补 FUPSI PIR 路线之外的空白）。

**PSR**：IBLT 最初为 set reconciliation 设计（GM11）。双方维护各自 IBLT，交换 delta 后直接 peeling 恢复差异。

### 7.2 Self-Healing（泄露后自愈）

```
  标准：K_{t+1} ← PRF(K_t)
  自愈：K_{t+1} ← PRF(K_t, H(X_{∪,t}))

  X_{∪,t} \ X_b 是对方独有元素 → 对攻击者不可预测
  → 对方下一次更新后密钥链自愈（1 轮内）
  → 零额外通信，利用并集输出自身的新鲜度
```

### 7.3 更多方向

| 方向 | 挑战 | 参考 |
|------|------|------|
| 恶意安全 | delta IBLT 完整性 + 恶意 OT/OPRF | PuGT26 |
| 多方扩展 | n-party UnionPeel + 通信复杂度 | GNT24 |
| 后量子安全 | LWE-based OT + LPN-based VOLE | YBH+25, RRT23 |
| Rateless IBLT | 动态 IBLT 容量，消除偶尔全量重建 | YGA24 |
